# AI 功能演示说明：风险识别与派单推荐智能辅助

## AI 功能名称
风险识别与派单推荐智能辅助

## 嵌入业务流程的位置
1. 老人/家属通过表单提交求助信息后
2. 系统自动调用 AI 引擎进行分析
3. 输出风险等级、事件类型、触发原因、置信度、建议处置和推荐责任人
4. 结果提交给网格员进行人工复核
5. 复核后生成工单

## 输入数据
- 今日状态（正常/轻微不适/紧急求助）
- 求助类型（无/健康/生活/维修/陪诊/其他）
- 求助文本描述
- 设备提醒（无/跌倒提醒/烟感提醒/水浸提醒/长时间未活动）
- 老人历史标签（健康标签、服务等级）
- 历史工单记录
- 人员资源和技能标签

## 输出结果
- **风险等级**：低风险 / 中风险 / 高风险
- **事件类型**：健康类 / 维修类 / 生活服务类 / 陪诊类 / 无异常
- **触发原因**：具体原因列表
- **AI 置信度**：0.60 ~ 0.98 的模拟置信度
- **建议处置**：根据风险等级给出处置建议
- **推荐责任人**：根据事件类型和风险等级推荐
- **推荐处理时限**：根据紧急程度设定
- **可解释规则说明**：状态规则、设备规则、文本规则、分类规则、综合规则

## 规则逻辑

### 风险等级判定
- **低风险**：状态为"正常"，设备提醒为"无"，描述无异常关键词
- **中风险**：状态为"轻微不适"，或描述含头晕/不舒服/两天/未报平安/家属反馈/轻微/乏力，或设备提醒为"长时间未活动"
- **高风险**：状态为"紧急求助"，或设备提醒为"跌倒提醒/烟感提醒"，或描述含跌倒/摔倒/昏迷/胸闷/联系不上/严重/突发/报警/呼吸困难

### 事件类型分类
- **健康类**：求助类型为健康，或描述含头晕/胸闷/不舒服/昏迷/跌倒/摔倒/呼吸困难
- **维修类**：求助类型为维修，或描述含漏水/断电/门锁/烟感/水浸/电路
- **生活服务类**：求助类型为生活，或描述含买药/代购/送餐/陪伴
- **陪诊类**：求助类型为陪诊，或描述含陪诊/去医院/复诊/挂号
- **无异常**：状态正常且无设备提醒

### AI 置信度计算
- 高风险强触发词越多，置信度越高
- 设备提醒与文本描述一致时，置信度提高
- 描述为空或过短时，置信度降低

### 建议处置
- **低风险**：进入常规巡访，系统留痕
- **中风险**：网格员电话核实，生成随访任务
- **高风险**：立即派单，通知家属和社区医生

### 推荐责任主体与处理时限

| 场景 | 推荐责任人 | 处理时限 |
|------|-----------|---------|
| 健康类 + 高风险 | 社区医生 + 网格员 | 30 分钟内响应 |
| 健康类 + 中风险 | 网格员（电话核实），必要时社区医生随访 | 2 小时内响应 |
| 健康类 + 低风险 | 系统留痕，纳入常规巡访 | 按巡访计划处理 |
| 维修类 | 物业/维修人员 | 4 小时内响应 |
| 生活服务类 | 志愿者或网格员 | 当日响应 |
| 陪诊类 | 志愿者 + 家属确认 | 1 日内安排 |
| 其他 | 网格员 | 按常规流程处理 |

### 可解释规则说明

每次 AI 识别后，系统自动生成五类可解释规则：

1. **状态规则**：根据今日状态（正常/轻微不适/紧急求助）说明触发或未触发状态风险规则
   - 示例：状态规则：今日状态为"紧急求助"，触发高风险规则

2. **设备规则**：根据设备提醒（跌倒提醒/烟感提醒/水浸提醒/长时间未活动）说明触发或未触发设备风险规则
   - 示例：设备规则：设备提醒为"跌倒提醒"，触发高风险规则

3. **文本规则**：根据描述文本中匹配到的关键词说明
   - 示例：文本规则：描述包含"跌倒、联系不上"等关键词

4. **分类规则**：根据求助类型说明事件分类依据
   - 示例：分类规则：求助类型为"健康"，识别为健康类事件

5. **综合规则**：汇总匹配的触发条件数量和最终 AI 置信度
   - 示例：综合规则：共匹配 3 条触发条件，AI 置信度为 95%

## 人工复核机制
此页面是"AI 不替代人工"的核心体现：
1. 展示 AI 识别结果供参考
2. 网格员可修改风险等级、事件类型
3. 网格员填写复核意见
4. 网格员决定是否生成工单
5. 系统记录 AI 原始判断和人工最终判断
6. 记录 `isAiModified` 字段（AI 是否被人工修改）
7. 中/高风险但不生成工单时弹出二次确认
8. 管理看板统计 AI 后人工修改次数

## 风险边界
- 系统不替代医生诊断
- 系统不输出确诊、治疗方案等医疗结论
- 高风险事件必须由社区工作人员人工复核
- 系统只提供优先级建议，不直接决定是否延迟服务
- 老人健康、住址、联系方式等敏感数据按角色分级授权
- 所有 AI 判断和人工修改均需留痕

## 敏感数据保护
- **不作为训练材料**：本原型中所有老人姓名、地址、电话、健康标签等个人信息仅用于本地功能演示，不作为任何 AI 模型的训练语料
- **数据脱敏**：电话默认显示 `138****0001` 格式，地址仅显示到楼栋，完整信息需授权确认后方可查看
- **本地存储**：所有数据仅存储在浏览器 localStorage 中，不上传至任何服务器
- **审计追踪**：所有关键操作均写入审计日志（`state.auditLogs`），可追溯操作角色、时间、类型和详情
- **导出管控**：数据导出为 JSON 文件，包含完整审计日志，便于合规审查

## AI 技术实现

系统**已默认接入 DeepSeek 大语言模型 API**（deepseek-chat），通过本地代理服务 `server/server.js` 安全调用：

1. **真实大语言模型**：优先使用 DeepSeek Chat API 进行风险识别与事件分类，结构化 prompt 引导模型输出 JSON
2. **规则引擎兜底**：当 LLM 不可用（网络异常、Key 错误、超时、返回格式异常）时，自动降级为内置规则引擎
3. **PII 脱敏**：`sanitizeRecordForLLM()` 在发送前移除姓名、电话、门牌号、紧急联系人等敏感信息
4. **来源透明**：AI 结果卡片明确标注 LLM / 规则引擎来源、模型名称和兜底状态

**重要**：无论使用何种 AI 方案，都必须保留人工复核入口和风险边界控制。
